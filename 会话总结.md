## 项目概况

这是一个 **Nuxt 4 全栈动态博客与轻量 CMS**：后台内容写入 PostgreSQL，前台通过 SSR 和 Nitro API 实时读取，无需重新构建。

当前工作目录：

```bash
/home/jiupan/blog-nuxt
```

核心能力：

- 博客前台：文章、分类、标签、归档、搜索、关于、友链和动态菜单。
- 管理后台：文章、分类、标签、菜单、站点设置、侧栏、图库和知识库管理。
- 用户系统：普通用户注册/登录与管理员权限。
- AI：摘要、写作辅助、SEO、链接检查、相关文章、语义搜索和站内问答。
- 知识库：文章及外部文件分块、Embedding、pgvector 检索和可选 Rerank。
- 文章对话：独立会话、历史记录、流式回答、重命名和删除。
- 展示能力：明暗主题、响应式布局、可配置页头/页脚/侧栏和媒体资源。

## 技术栈与硬约束

```text

Docker Compose / Nginx
```

- Node.js 使用 `.nvmrc` 指定的版本，当前为 Node.js 22。
- Prisma 保持在 6.19.x，不能直接升级到 Prisma 7。
- PostgreSQL 使用 `pgvector/pgvector:pg16`；向量字段为 `vector(1536)`。
- `nuxt.config.ts` 必须保留 `srcDir: 'app'`。
- 数据库结构变更必须同时提交 Prisma migration，不能只修改 `schema.prisma`。

## 路由与功能边界

主要前台路由：

```text
/                         首页
/posts                    文章文库
/[slug]                   文章详情
/categories/[slug]        分类文章
/tags/[slug]              标签文章
/archive                  归档
/about                    关于
/link                     友链
/lab                      AI 实验室
/tools                    工具箱
/login、/register         普通用户认证
```

文章详情使用根级 `/{slug}`，链接应通过 `app/utils/post-path.ts` 生成，不要恢复旧的 `/posts/[slug]` 页面路径。

后台位于 `/admin/**`，包含文章、分类、标签、菜单、设置、侧栏、图库和知识库页面；后台页面通过 `app/middleware/admin-auth.ts` 保护，并在 `nuxt.config.ts` 中关闭 SSR。

API 按以下边界组织：

```text
/api/posts、categories、tags、archive、search、settings、menus   公开内容
/api/auth/**                                                     登录、注册、退出、会话
/api/ai/**                                                       登录用户 AI 工具
/api/posts/[slug]/chat/**                                        文章对话
/api/admin/**                                                    管理员接口
/uploads/**                                                      上传文件访问
```

所有 `/api/ai/**` 都通过 `withAiUsage` 或 `requireLabUser` 要求有效用户会话并记录用量。所有 `server/api/admin/**/*.ts` 必须调用 `requireAdmin(event)`，已有测试扫描这一约束。

## 架构约束

`架构治理计划.md` 的阶段 0 到阶段 6 已完成，继续沿用以下分层：

```text
app/pages、app/components       UI 编排与交互
app/composables                 页面状态、表单状态、API 调用编排
server/api                     鉴权、参数校验、调用 service、返回响应
server/services                业务规则与跨接口复用
server/utils                   通用服务端基础设施
types/api.ts、types/dto/**      前后端共享契约
```

关键规则：

- 公开文章条件统一为 `PUBLISHED` 且 `publishedAt <= now`，查询集中在 `post-query.service.ts`。
- 文章写入、分类标签、菜单、设置、媒体和权限规则分别留在对应 service 中，不要堆回 API handler。
- RAG raw SQL 只允许出现在 `server/services/rag/post-chunk.repository.ts`，测试约束 raw SQL 边界。
- 知识文件支持 MD、TXT、PDF 和 DOCX；处理逻辑位于 `server/services/knowledge-files/**`。
- Markdown 禁用原始 HTML，渲染结果再经 `sanitize-html` 清洗。
- 公开内容使用 HTTP 短缓存响应头，不使用 Nitro server cache。

## 数据与存储

主要 Prisma 领域：

```text
用户：User、AiUsageLog
内容：Post、Category、Tag、PostTag、PostRelation
文章对话：ArticleChatConversation、ArticleChatMessage
菜单与设置：Menu、MenuItem、Setting
知识库：PostChunk、KnowledgeDocument、KnowledgeSyncJob
外部知识：KnowledgeFile、KnowledgeFileChunk
查询审计：RagQueryLog
```

关键状态：

- 用户角色为 `ADMIN` / `USER`，账号状态为 `ACTIVE` / `DISABLED`。
- 文章状态为 `DRAFT` / `PUBLISHED` / `ARCHIVED`。
- 知识文档支持待同步、同步中、已同步、过期、失败和禁用状态。

评论由独立的 Twikoo 服务管理，不写入博客 Prisma 数据库。尚未正式模型化：友链数据、访问统计、文章修订历史和对象存储。

历史风险：本地数据库曾因早期 init migration 与实际库状态不一致出现 drift。遇到 Prisma 要求 reset 时，不要重置生产库或有价值的数据；先核对 migration 历史和实际 schema，再决定 `prisma migrate resolve` 或手工迁移。

## 认证、安全与上传

- Session 使用 `nuxt-auth-utils`，Cookie 为 `__Host-nuxt-session`，7 天有效，开启 `secure`、`httpOnly` 和 `sameSite=lax`。
- 登录支持用户名或邮箱；登录与注册都使用内存级固定窗口限流。该限流不跨实例，多实例部署需改为共享存储。
- 普通用户权限走 `requireUser`，管理员权限走 `requireAdmin`。
- 上传和删除路径必须经过 `server/services/media/path-safety.ts`。
- 一般上传最大 50MB；头像视频最大 5MB；知识文件请求限制为 11MB。
- 图片由 Sharp 校验和优化；Favicon 可使用经过安全检查的 SVG；GIF 表情包保留动画。
- 上传目录来自 `UPLOAD_DIR`，知识文件目录来自 `KNOWLEDGE_FILE_DIR`，生产环境必须持久化两者。

设置边界集中在：


需要特别注意：

- `sidebar_modules` 保存侧栏模块的顺序和显隐。

## AI 现状与后续路线

2026-07-15 已完成一次代码级盘点。当前判断：AI 功能覆盖面较完整，适合个人博客和小规模使用；生产可靠性、RAG 评测和可观测性仍需加强。

当前能力分为三类：

- 模型生成：文章摘要、SEO 元信息与建议、写作助手、相关文章推荐、知识库问答。
- RAG：文章与知识文件分块、Embedding、向量与关键词混合检索、RRF、可选 Rerank、引用和文章内流式对话。
- 规则工具：站点洞察、草稿整理、月度回顾、外链检查及 SEO 基础评分；这些主要是确定性规则，不应误认为都调用了大模型。

已有优势：

- AI 输出使用 Zod schema 和 parser 校验，文章、分类、标签及引用 ID 会二次过滤。
- 知识库支持文章以及 MD、TXT、PDF、DOCX 文件，并有同步状态、任务记录、检索测试和查询日志。
- AI Lab 要求登录并记录用量；文章对话支持登录用户和匿名身份隔离。
- RAG 已实现向量与关键词融合、来源去重、上下文长度限制和可选 Rerank。

已知问题，按优先级排列：

1. **同步任务可靠性（2026-07-15 已完成第一版）**：已移除请求内 `setImmediate()`，改为数据库持久化队列和 Nitro Worker；具备多实例唯一任务键、乐观认领、心跳、租约超时恢复、三次指数退避重试和优雅停止。部署流程已调整为先迁移再启动应用。未来如果任务规模明显增大，可再拆成独立 Worker 容器。
2. **AI 请求韧性**：Chat、Embedding、Rerank 客户端缺少统一超时、429/5xx 重试、指数退避、最大输出 Token、Token/成本采集和完整流式解析容错。
3. **流式引用准确性**：流式问答目前在非拒答时直接引用前三个上下文来源，并不代表模型真正使用了这些来源；应输出并验证引用编号。
4. **检索降级与阈值**：Embedding 失败时关键词搜索也无法执行；Rerank 失败会静默降级；缺少最低相关性阈值，低质量 Top-K 仍可能进入模型。
5. **多轮检索改写**：文章对话主要拼接上一条用户问题，复杂指代追问召回不稳定；需要轻量 Query Rewriter。
6. **流式失败持久化**：非流式失败会保存 `FAILED` 助手消息，流式失败可能只留下用户消息。
7. **评测不足**：缺少真实 PostgreSQL/pgvector 集成测试和固定 RAG 问答评测集，当前无法量化 Recall@K、引用准确率、拒答准确率和回答忠实度。
8. **外链检查安全**：只校验 URL 中显式的本地/私网主机，没有验证 DNS 解析结果和每次重定向目标，仍有 SSRF 加固空间。
9. **可观测性不足**：查询日志未充分记录召回片段、检索/模型分段耗时、Token、模型版本、费用和用户反馈。

推荐实施顺序：

```text
阶段一：可靠任务队列（已完成第一版） -> AI 客户端超时/重试 -> 准确流式引用 -> 检索降级/阈值 -> 外链检查加固
阶段二：RAG 评测集 -> Token/成本/延迟指标 -> 用户反馈闭环 -> 后台质量看板
阶段三：编辑器局部改写与 Diff -> 定期内容体检 -> 内容缺口选题 -> 相关推荐增量刷新
```

如果只做三项，优先选择：

1. ~~将知识库同步迁移到可靠任务队列，并支持失败重试和死任务恢复。~~ 已完成第一版。
2. 建立 30～50 个问题的 RAG 评测集，覆盖可回答、跨文章、同义、多轮和应拒答场景。
3. 改造流式回答协议，让模型返回可验证的引用编号并在 SSE `done` 事件中输出最终引用。
