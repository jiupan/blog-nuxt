# Nuxt 博客自研评论系统设计方案（当前项目修订版）

> **状态：已废弃。** 项目已于 2026-07-18 切换为 Twikoo。本文仅保留为历史设计记录，不再作为当前实现依据；现有自研评论数据表暂时保留用于安全回退，应用已停止读写。

> 本文以当前仓库为实施基线，适用于现有 Nuxt 4、Nitro Server API、Prisma 6、PostgreSQL、`nuxt-auth-utils` 和 `nuxt-security` 架构。
>
> 修订日期：2026-07-15。

## 一、结论与实施边界

这套评论系统可以在当前项目中实现，核心架构不需要推翻。现有项目已经具备：

- Nuxt 4 页面和 Nitro Server API。
- PostgreSQL 与 Prisma。
- 登录用户、管理员角色和服务端 Session。
- Zod 参数校验、统一 API 响应和错误结构。
- `markdown-it`、`sanitize-html` 等内容处理依赖。
- 带任务抢占、心跳、重试和超时恢复的知识库 Worker，可作为通知 Worker 的实现参考。
- 文章详情页和后台导航均有清晰的评论功能接入位置。

第一版应以“安全发布、可审核、计数一致”为目标。公开评论写接口上线时，参数校验、服务端 Markdown 清洗、基础限流、幂等和 Turnstile 必须同时启用，不能先开放写入再补安全能力。

## 二、第一版功能范围

### 2.1 必须实现

- 匿名访客和登录用户发表评论。
- 两层展示、完整保存回复关系。
- 最新、最早、最热排序。
- 点赞与取消点赞。
- 举报。
- 后台审核、拒绝、标记垃圾、软删除、恢复和置顶。
- 后台回复。
- 文章可见评论总数和一级评论回复数。
- 签名匿名身份。
- Redis 限流与幂等控制。
- Cloudflare Turnstile。
- 敏感词及基础风险规则。
- 独立的评论 Markdown 渲染与 XSS 清洗。
- 审核操作日志。
- Outbox 邮件通知和失败重试。
- 移动端适配。

### 2.2 暂不实现

- 无限层级树形展示。
- WebSocket 实时评论。
- 评论图片上传。
- 匿名用户自行编辑或删除评论。
- 用户积分、等级、勋章和表情商店。
- 点赞邮件通知。
- AI 自动发布回复。
- 评论列表的共享缓存。
- 将全部评论直接写入 RAG。

服务端继续遵循当前项目约定：

## 四、回复结构

数据库保存完整的实际回复关系，前端只平铺展示两层：

```text
张三：这篇文章写得很好
  ├── 李四 回复张三：确实不错
  ├── 王五 回复李四：特别是 Redis 那部分
  └── 作者 回复王五：后面会继续补充
```

字段含义：

```text
rootId          所属一级评论；一级评论为 null
parentId        实际被回复的评论
replyToUserId   被回复的登录用户；匿名对象为 null
```

回复匿名评论时，展示名称和通知邮箱从 `parent` 评论读取，不能只依赖 `replyToUserId`。

服务端计算规则：

```text
parent.rootId 为空：rootId = parent.id
parent.rootId 非空：rootId = parent.rootId
parent.userId 非空：replyToUserId = parent.userId
```

必须验证父评论属于当前文章，且状态允许回复。客户端不能提交 `rootId` 和 `replyToUserId`。

## 五、数据库设计

### 5.1 类型和关系原则

当前项目的 `User.id` 和 `Post.id` 均为 `Int`，评论相关主键和外键必须继续使用 `Int`，不能使用 `BigInt`。

需要在现有模型补充反向关系：

```prisma
model User {
  // 现有字段保持不变
  comments                     Comment[]             @relation("CommentUser")
  replyTargetComments          Comment[]             @relation("CommentReplyToUser")
  commentLikes                 CommentLike[]
  commentReports               CommentReport[]       @relation("CommentReportUser")
  handledCommentReports        CommentReport[]       @relation("CommentReportHandler")
  commentNotifications         CommentNotification[] @relation("CommentNotificationReceiver")
  commentAuditLogs             CommentAuditLog[]     @relation("CommentAuditOperator")
}

model Post {
  // 现有字段保持不变
  comments          Comment[]
  commentCount      Int               @default(0)
  commentReviewMode CommentReviewMode @default(DEFAULT)
}
```

项目当前是单作者博客，第一版约定 `ADMIN` 即文章作者。未来支持多作者时，再给 `Post` 增加 `authorId`。不要在当前没有作者关系的情况下实现虚假的文章作者权限。

`commentReviewMode` 已包含 `DISABLED`，不再额外增加 `commentEnabled`，避免两个字段产生冲突。

```prisma
enum CommentReviewMode {
  DEFAULT
  DIRECT
  REVIEW
  DISABLED
}
```

### 5.2 Comment

```prisma

  @@index([status, createdAt])
}

  PENDING
  APPROVED
  REJECTED
  SPAM
  DELETED
}

enum CommentSource {
  WEB
  ADMIN
  API
}
```

规则：

- 登录评论使用 `userId`；匿名评论使用 `visitorHash`；两类评论都保存服务端生成的 `actorKey`，用于幂等和“仅本人可见”查询。
- 数据库迁移中增加约束，保证 WEB 评论恰好具有一种身份。
- `content` 保存原始 Markdown，`contentHtml` 只保存服务端清洗后的 HTML。
- `isAuthor` 由服务端根据管理员身份计算，客户端不能提交。
- 评论采用软删除，不单独物理删除树中节点。
- 删除文章时允许级联清理评论；评论之间不使用级联删除，避免回复链断裂。

### 5.3 CommentLike

PostgreSQL 对包含 `NULL` 的唯一约束允许多条空值记录，因此不能使用两个可空字段的复合唯一约束来防止重复点赞。

统一生成身份键：

```text
登录用户：actorKey = user:{userId}
匿名用户：actorKey = visitor:{visitorHash}
```

`actorKey` 必须由服务端生成，不能接收客户端值。

```prisma
model CommentLike {
  id          Int      @id @default(autoincrement())
  commentId   Int
  actorKey    String   @db.VarChar(80)
  userId      Int?
  visitorHash String?  @db.VarChar(64)
  createdAt   DateTime @default(now())

  comment     Comment  @relation(fields: [commentId], references: [id], onDelete: Cascade)
  user        User?    @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([commentId, actorKey])
  @@index([commentId, createdAt])
}
```

迁移中增加约束，保证 `userId` 与 `visitorHash` 恰好一个非空。

通知 Worker 复用现有知识库 Worker 的模式：

- 原子抢占一条任务。
- `lockedBy`、`lockedAt` 和心跳续租。
- 指数退避重试。
- 回收超时任务。
- Nitro 关闭时停止轮询。
- 多实例下只能由成功抢占的 Worker 发送邮件。

第一版只发送：回复通知、新评论通知、审核通过通知。点赞不发送邮件。

## 六、匿名身份和隐私数据

第一次访问评论接口时，服务端生成随机 UUID，并设置：

```text
comment_visitor = visitorId.signature
```

Cookie 配置：

```text
HttpOnly=true
Secure=生产环境启用
SameSite=Lax
Path=/
MaxAge=365天
```

签名：

```text
signature = HMAC-SHA256(visitorId, COMMENT_VISITOR_SECRET)
visitorHash = HMAC-SHA256("visitor:" + visitorId, COMMENT_HASH_SECRET)
```

数据库不保存原始 Cookie 值，只保存 `visitorHash`。

隐私字段处理：

- 邮箱使用带随机 IV 和认证标签的 AES-256-GCM 加密，密文包含密钥版本以支持轮换。
- `emailHash` 使用带 `email:` 前缀的服务端密钥 HMAC，不使用可离线枚举的裸 SHA-256。
- `ipHash` 使用带 `ip:` 前缀的服务端密钥 HMAC，不保存原始 IP。
- 邮箱、IP Hash、访客 Hash 不得出现在公开 API。
- 对日志和隐私字段设置保留期限。
- 如果使用 Gravatar，应作为显式配置并评估邮箱 Hash 对外暴露问题；默认建议使用本地生成头像。

匿名评论字段：昵称必填，邮箱是否必填由全局配置控制，网站选填且只接受 `http` 或 `https`。

## 七、API 设计

当前公开文章接口以 `slug` 为路由参数，因此评论接口保持一致。

```text
GET    /api/posts/{slug}/comments
POST   /api/posts/{slug}/comments
GET    /api/comments/{rootId}/replies
POST   /api/comments/{commentId}/like
DELETE /api/comments/{commentId}/like
POST   /api/comments/{commentId}/reports
DELETE /api/comments/{commentId}
```

后台接口：

```text
GET  /api/admin/comments
POST /api/admin/comments/batch
POST /api/admin/comments/{commentId}/approve
POST /api/admin/comments/{commentId}/reject
POST /api/admin/comments/{commentId}/spam
POST /api/admin/comments/{commentId}/restore
POST /api/admin/comments/{commentId}/pin
POST /api/admin/comments/{commentId}/reply
GET  /api/admin/comments/{commentId}/context
```

所有响应继续使用当前统一结构：

```json
{
  "code": 0,
  "message": "成功",
  "data": {}
}
```

权限规则：

- 游客可以查看、匿名发布、点赞和举报，不能删除评论。
- 登录用户可以发布、点赞、举报、查看自己的待审核评论，并软删除自己的评论。
- 当前项目中的 `ADMIN` 视为站点作者，可以执行全部后台操作。
- 所有权限均在 Service 层根据 Session 和服务端身份判断，不能只依赖前端隐藏按钮。

### 7.1 评论列表

```http
GET /api/posts/{slug}/comments?page=1&pageSize=10&sort=latest
```

`sort` 支持 `latest`、`oldest`、`hottest`，`pageSize` 最大 50。

列表只分页一级评论。每个一级评论默认返回最早 3 条可见回复；剩余回复通过独立接口分页加载，避免单个热门评论拖垮整个响应。

返回中可以包含当前身份的 `liked`，以及当前身份自己提交的 `PENDING` 评论，因此必须：

```text
Cache-Control: no-store
```

不能使用文章详情接口的 public 缓存策略。

删除占位规则：

- `APPROVED` 评论正常返回。
- `DELETED` 一级评论存在可见回复时返回占位节点。
- `DELETED` 且没有可见回复时不返回。
- 占位节点的内容固定为“该评论已删除”，不能返回原内容和个人信息。

### 7.2 提交评论和回复

```http
POST /api/posts/{slug}/comments
X-Request-Id: UUID
Content-Type: application/json
```

```json
{
  "content": "文章写得很好",
  "parentId": null,
  "nickname": "张三",
  "email": "example@example.com",
  "website": "https://example.com",
  "turnstileToken": "..."
}
```

登录用户的名称和邮箱从 Session / User 读取，忽略匿名身份字段。`rootId`、`replyToUserId`、`status`、`isAuthor`、`riskScore` 均不得由客户端提交。

处理顺序：

```text
解析并验证文章 slug
→ 检查文章已发布及评论模式
→ 解析 Session 或签名匿名 Cookie
→ 校验 Origin、Content-Type 和请求大小
→ Redis 多维限流
→ 按策略验证 Turnstile
→ Zod 参数校验
→ 验证父评论及所属文章
→ 幂等和重复内容检查
→ 评论专用 Markdown 渲染与清洗
→ 敏感词和风险计算
→ 确定状态
→ 数据库事务写评论、计数、审计和 Outbox
→ 返回结果
```

返回直接通过或待审核的状态。待审核评论只对同一 `userId` 或 `visitorHash` 可见，不能仅依赖前端内存，否则刷新后会消失。

### 7.3 点赞

只能点赞 `APPROVED` 评论。

点赞事务：

1. 使用服务端生成的 `actorKey` 插入 `CommentLike`。
2. 只有插入成功时才将 `likeCount` 原子加一。
3. 唯一冲突返回“已点赞”，不能再次加一。

取消点赞事务：

1. 按 `commentId + actorKey` 执行 `deleteMany`。
2. 只有删除行数为 1 时才将 `likeCount` 原子减一。
3. 数据库增加 `likeCount >= 0` 约束。

### 7.4 举报

只能举报公开可见评论。同一 `actorKey` 对同一评论只能举报一次。创建举报和增加 `reportCount` 必须在同一事务中。

## 八、审核状态与计数一致性

`Post.commentCount` 定义为该文章所有 `APPROVED` 评论数量，包含一级评论和回复。

`Comment.replyCount` 只在一级评论上使用，定义为该根评论下所有 `APPROVED` 回复数量。

状态转换统一由 `comment-moderation.service.ts` 处理：

```text
非 APPROVED → APPROVED
  Post.commentCount + 1
  如果是回复：RootComment.replyCount + 1

APPROVED → 非 APPROVED
  Post.commentCount - 1
  如果是回复：RootComment.replyCount - 1

APPROVED → APPROVED
非 APPROVED → 非 APPROVED
  不修改计数
```

创建时直接通过的评论，也必须在创建事务中增加计数。

每次状态变化同时写入 `CommentAuditLog`。审核接口必须使用带旧状态条件的 `updateMany` 或事务内锁，防止重复点击、并发审核导致计数重复变化。

迁移中为四个计数字段增加非负约束：

```text
Post.commentCount >= 0
Comment.likeCount >= 0
Comment.replyCount >= 0
Comment.reportCount >= 0
```

后台提供计数校准任务，从真实状态重新计算计数，用于修复异常或历史数据。

## 九、评论内容安全

不能直接复用当前文章 Markdown 渲染器。文章渲染器允许图片、标题、表格和部分样式，并包含服务端 Shiki 高亮；评论需要独立的轻量渲染器：

```text
server/services/comment/comment-render.service.ts
```

要求：

- `markdown-it` 设置 `html: false`。
- 原文长度 2～3000 字，渲染后 HTML 也设置上限。
- 只允许 `p`、`br`、`strong`、`em`、`del`、`code`、`pre`、`blockquote`、`ul`、`ol`、`li`、`a`。
- 禁止图片、标题、表格、SVG、表单、音视频、iframe 和所有 style/event 属性。
- 链接只允许 `http`、`https`，统一添加 `nofollow noopener noreferrer` 和 `_blank`。
- 限制外链数量，禁止纯链接评论。
- 代码块只转义或使用轻量客户端高亮，避免大代码块触发昂贵的服务端 Shiki 处理。
- 前端只有 `contentHtml` 可以通过 `v-html` 渲染，且该字段只能来自服务端清洗结果。

## 十、限流、Turnstile 与防重复提交

### 10.1 Redis 限流

当前项目的内存 `Map` 限流可继续用于登录和注册的基础保护，但评论系统必须使用 Redis，保证重启和多实例下一致。

建议默认值：

```text
IP：1 分钟 3 条，1 小时 15 条，1 天 50 条
visitorHash：1 分钟 2 条，1 小时 10 条
登录用户：1 分钟 5 条，1 小时 30 条
点赞：同一身份 1 分钟 30 次
举报：同一身份 1 小时 10 次
```

使用 Lua 或等价原子命令同时完成计数和过期设置。达到限制时返回现有 `RATE_LIMITED` 错误和 `retryAfterSeconds`。

生产 Nginx 已设置 `X-Real-IP` 和 `X-Forwarded-For`。部署时必须确保应用只能经过可信反向代理访问，且 Nginx 覆盖外部传入的转发头，避免伪造 IP 绕过限流。

### 10.2 Turnstile

以下场景强制验证：

- 匿名用户首次评论。
- 评论包含链接。
- 短时间连续提交。
- 风险评分较高。
- 最近存在多次失败。

服务端调用 Turnstile Siteverify，校验 token、hostname 和 action，并设置网络超时。生产默认 fail closed；开发环境可通过显式配置禁用，不能因缺少密钥静默绕过。

### 10.3 幂等与重复内容

客户端每次提交生成 `X-Request-Id`。Redis 保存：

```text
comment:idempotency:{actorKey}:{requestId}
```

过期时间 10 分钟。成功后保存评论 ID，重复请求返回同一结果。

同时计算：

```text
contentHash = HMAC-SHA256(postId + actorKey + parentId + normalizedContent)
```

短时间内提交相同内容直接拒绝。不能只依赖前端禁用提交按钮。

## 十一、风险规则和审核模式

基础规则：

- 重复字符、纯链接、外链数量和 Emoji 数量限制。
- 广告词、敏感词。
- 同一身份短时间跨文章重复内容。
- 首次匿名身份。
- Turnstile 结果。
- 登录用户和历史审核记录。

建议初始状态策略：

```text
ADMIN 发布：APPROVED
文章 REVIEW：PENDING
文章 DIRECT：低风险 APPROVED，高风险 PENDING/SPAM
文章 DEFAULT：使用全局模式
文章 DISABLED：拒绝提交
首次匿名评论：默认 PENDING
命中明确垃圾规则：SPAM
```

风险阈值、全局审核模式和敏感词通过现有 `Setting` 体系管理，不写死在 Service 中。

建议增加的设置键：

```text
comment_review_mode
comment_anonymous_enabled
comment_email_required
comment_turnstile_mode
comment_risk_approve_max
comment_risk_spam_min
comment_sensitive_words
comment_admin_notification_email
```

这些设置不加入公开设置白名单。浏览器需要的 Turnstile Site Key 通过 `runtimeConfig.public` 暴露。

## 十五、缓存策略

第一版评论列表不做共享缓存，原因是响应包含当前身份的 `liked` 和待审核评论。

```text
评论列表：no-store
评论回复：no-store
文章详情：继续使用现有 public 缓存
评论数量：存储在 Post.commentCount
后台接口：no-store
```

如果未来需要缓存，应拆分“公开评论数据”和“当前身份状态”，不能把带用户状态的响应放入 CDN 公共缓存。

## 十九、最终架构原则

1. 使用当前项目的 `Int` 主键体系。
2. 数据库保存完整回复关系，前端只展示两层。
3. 匿名身份使用签名 Cookie，数据库只存 Hash。
4. 点赞和举报使用服务端 Actor Key 保证唯一性。
5. 评论内容由独立的服务端渲染器解析和清洗。
6. 所有状态变化和冗余计数在事务内统一处理。
7. 删除使用软删除，保留回复链。
8. 用户相关评论响应不进入公共缓存。
9. 写接口上线时同时具备限流、幂等、验证码和审核。
10. 邮件发送通过带租约和重试的 Outbox Worker 解耦。
11. 所有审核操作保留日志。
12. AI 只能提供建议，不能成为唯一审核依据。
