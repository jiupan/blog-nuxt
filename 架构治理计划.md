# 架构治理计划

本文档用于跟踪 Nuxt 动态博客项目的代码结构治理进度。目标不是一次性大重构，而是按风险和收益分阶段降低技术债，让后续功能扩展更稳。

## 状态标记

| 标记 | 含义 |
| --- | --- |
| `[未实现]` | 尚未开始 |
| `[进行中]` | 正在处理，尚未完成验收 |
| `[已实现]` | 已完成并通过验收 |
| `[暂缓]` | 当前阶段不做，保留原因 |

## 治理原则

- 优先处理会阻碍后续重构的基础设施：类型检查、测试、脚本、契约。
- 优先拆“大文件”和“高变化模块”，不追求目录形式上的完美。
- API handler 保持薄：鉴权、参数校验、调用 service、返回响应。
- 业务规则放 service，数据库细节放 repository 或明确的 data access 函数。
- 每个阶段都要能独立合并，不做跨几十个文件的大爆炸式重构。
- 保留当前 Nuxt 4 + Prisma + PostgreSQL + pgvector 技术栈，不在治理阶段顺手换框架。

## 总体路线

```text
阶段 0：建立质量门禁
  ↓
阶段 1：拆解前端大文件
  ↓
阶段 2：重整 AI 服务边界
  ↓
阶段 3：沉淀核心领域 service
  ↓

  ↓
阶段 5：整理设置、菜单、媒体等可增长模块
  ↓
阶段 6：补齐测试覆盖和重构回归保护
```
