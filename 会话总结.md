## 一、用户目标

用户希望基于当前目录中的 `实现方案.md` 做一个个人博客系统。核心诉求是：

```text
后台发文章 -> 写入数据库 -> 前台 SSR 实时读取 -> 不需要重新构建
```

因此项目采用 **Nuxt 4 全栈动态博客**，不是静态生成博客。

当前用户正在优化前台首页，希望视觉接近一张 HEO 风格博客首页 UI 设计稿，但内容必须来自当前系统真实数据，不能把设计稿中的文章、分类、标签、分页、统计、作者信息直接写死。

## 十、重要问题和处理记录

### 1. `app/` 目录没有被 Nuxt 识别

最初页面返回 Nuxt Welcome，是因为 Nuxt 没有按当前项目结构识别 `app/` 作为源码目录。

已在 `nuxt.config.ts` 加入：

```ts
srcDir: 'app'
```

### 2. 前台 SWR 缓存导致旧页面不刷新

原方案中给前台页面加了 SWR 缓存。开发过程中发现它会缓存旧 HTML，并且在本地触发 `.nuxt/cache/nuxt/payload` 的目录读写错误。

已移除前台页面的 SWR routeRules，只保留：

```ts
routeRules: {
  '/admin/**': { ssr: false },
  '/api/admin/**': { cors: false }
}
```

这更符合“发文章后前台实时可见”的目标。

### 3. 不要用 PowerShell 批量重写中文源文件

之前用 PowerShell 批量替换导入路径时，部分中文源码被写成乱码。后来已修复。

后续编辑中文文件时，优先使用：

```text
apply_patch
```

不要用 `Set-Content` 批量重写包含中文的源码。

### 4. Nuxt build 的网络告警

构建时 Nuxt Fonts / Nuxt Icon 可能尝试访问：

```text
https://fonts.google.com/metadata/fonts
https://fonts.google.com/metadata/icons
```

网络不通时会有 warning，但提升权限后的构建最终通过。

### 5. Prisma 版本

不要直接升级到 Prisma 7。当前项目使用 Prisma 6.19.3，更符合当前 `schema.prisma` 写法。

### 6. 当前目录不是 Git 仓库

当前目录执行：

```bash
git status --short
```

结果：

```text
fatal: not a git repository (or any of the parent directories): .git
```

因此不能依赖 Git diff/status 来检查改动。

### 7. 内置浏览器不可用

尝试使用 Browser 插件连接内置浏览器做截图检查时，浏览器列表为空：

```text
Browser is not available: iab
```

## 十五、2026-06-28 远程服务器 Docker 部署完整方案

### 部署架构

采用 **本地构建 → 导出 tar → 上传服务器 → 加载运行** 模式，服务器不需要承担构建的内存压力：

```
本地（WSL + Docker Desktop）      远程服务器（阿里云 ECS，低内存）
─────────────────────           ─────────────────────────
docker build -t blog-app .
docker save -o blog-app.tar
    ↓
scp blog-app.tar ──────────→  docker load -i blog-app.tar
scp docker-compose.server.yml → docker compose up -d
```

### 项目文件清单

| 文件 | 用途 |
|------|------|
| `Dockerfile` | 多阶段构建（deps → builder → runner），Node 22 Alpine |
| `docker-compose.yml` | 本地开发用（含 `build: .`） |
| `docker-compose.server.yml` | 服务器用（用 `image: blog-app:latest`，不用 build） |
| `.dockerignore` | 排除 `node_modules`、`.output`、`.data`、`uploads`、`.git` |
| `.gitignore` | 追加 `blog-app.tar` 忽略 |

### `.dockerignore` — 减小构建上下文

```dockerignore
node_modules
.output
.data
uploads
.git
.gitignore
.env
.env.example
*.md
LICENSE
```

### `docker-compose.server.yml` — 服务器 compose 文件

### 手动创建管理员账号

seed 脚本依赖 `tsx` 和 `@prisma/client` 模块解析问题，在服务器上直接用 Node 一行脚本创建：

```bash
docker compose -f docker-compose.server.yml exec app node -e "
const { PrismaClient } = require('./node_modules/@prisma/client');
const argon2 = require('./.output/server/node_modules/argon2');
(async () => {
  const prisma = new PrismaClient();
  const hash = await argon2.hash('你的密码', { type: argon2.argon2id });
  await prisma.user.upsert({
    where: { username: 'admin' },
    update: { passwordHash: hash },
    create: { username: 'admin', passwordHash: hash }
  });
  console.log('Admin user created');
  await prisma.\$disconnect();
})();
"
```

> 注意：bash 中 `$disconnect` 需要转义为 `\$disconnect`。

### 两个关键 nuxt.config.ts 配置

**1. 关闭 CSP 强制 HTTPS 升级**

`nuxt-security` 默认的 `upgrade-insecure-requests` 会让浏览器把 HTTP 请求自动升级为 HTTPS，HTTP 部署时会导致 SSL 错误：

```ts
security: {
  headers: {
    contentSecurityPolicy: {
      'upgrade-insecure-requests': false
    }
  }
}
```

**2. 允许 HTTP 连接发送 session cookie**

`nuxt-auth-utils` 在 `NODE_ENV=production` 时默认 `secure: true`，cookie 只在 HTTPS 下发送，导致登录成功但 cookie 不生效。**配置必须放在 `runtimeConfig` 里面**（不是顶层 `session`）：

```ts
runtimeConfig: {
  session: {
    cookie: {
      secure: false
    }
  },
  // ... 其他配置
}
```

### Nginx 反向代理

Nuxt 监听 `0.0.0.0:3000`，Nginx 代理 80/443 到它：

```bash
sudo apt update && sudo apt install nginx -y

sudo tee /etc/nginx/sites-available/blog << 'NEOF'
server {
    listen 80;
    server_name blog.ittx.cn;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
NEOF

sudo ln -sf /etc/nginx/sites-available/blog /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx
```

### HTTPS（Certbot）

```bash
sudo apt install certbot python3-certbot-nginx -y
sudo certbot --nginx -d blog.ittx.cn
```

Certbot 自动获取证书并配置 301 跳转（HTTP → HTTPS），证书自动续期。

**阿里云安全组需加 `TCP 443` 入站规则。**

### HTTPS 下 nuxt.config.ts 配置

HTTPS 部署后，需恢复两项默认值：

```ts
// 恢复 CSP 强制升级（允许浏览器自动 HTTPS）
security: {
  headers: {
    contentSecurityPolicy: {
      'upgrade-insecure-requests': true
    }
  }
},
// 移除 runtimeConfig.session（使用默认 secure: true，cookie 仅 HTTPS 下发）
```

这也是当前项目的最终配置状态。

## 十七、nuxt-security 频率限制

`nuxt-security` 默认开启 rate limiter，频繁访问返回 `429 Too Many Requests`。

**关闭方式** —— 在 `nuxt.config.ts` 的 `security` 里加：

```ts
security: {
  rateLimiter: false,
  // ...
}
```

如果以后需要限频，可改为自定义阈值而非完全关闭。
