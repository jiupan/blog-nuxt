## 五、必须修复的会话权限问题

这是整个功能中最关键的一点。

当前 `requireAdmin()` 直接信任 Cookie Session 中保存的角色：

```ts
if (user.role !== 'ADMIN') {
  throw forbidden()
}
```

这意味着：

1. 用户登录时是管理员。
2. 另一个管理员将其降为普通用户或禁用。
3. 原 Session 中仍然保存着 `ADMIN`。
4. 在 Session 过期前，他可能仍然能调用后台接口。

所以不能只做管理页面，必须同时改变鉴权逻辑。

推荐让 `requireUser()` 根据 Session 中的用户 ID 查询数据库：

```ts
export async function requireUser(event: H3Event) {
  const session = await requireUserSession(event)
  const sessionUser = session.user as SessionUser | undefined

  if (!sessionUser?.id) {
    throw unauthorized()
  }

  const user = await prisma.user.findUnique({
    where: { id: sessionUser.id },
    select: {
      id: true,
      username: true,
      email: true,
      role: true,
      status: true
    }
  })

  if (!user) {
    throw unauthorized()
  }

  if (user.status !== 'ACTIVE') {
    throw forbidden('账号已被禁用')
  }

  return user
}
```

然后 `requireAdmin()` 判断数据库返回的最新角色：

```ts
export async function requireAdmin(event: H3Event) {
  const user = await requireUser(event)

  if (user.role !== 'ADMIN') {
    throw forbidden()
  }

  return user
}
```

这样：

- 禁用用户立即生效
- 降低角色立即生效
- 删除不存在的用户立即失效
- 不需要等待最长 7 天的 Session 过期

代价是每次受保护请求多一次用户查询。对于博客后台和当前用户规模，这是可接受且最可靠的方案。

如果以后请求量很大，可以再引入短期缓存或 `sessionVersion`，现在没必要提前复杂化。

## 六、管理员安全规则

建议把以下规则设为硬约束，而不是只在前端禁用按钮。

### 管理员不能操作自己

当前管理员不能：

- 禁用自己
- 将自己降为普通用户
- 删除自己

可以允许修改自己的密码，但建议后续放入独立的“账号安全”页面。

### 始终保留一个正常管理员

系统必须保证：

```text
role = ADMIN AND status = ACTIVE
```

的用户至少有一个。

修改角色或状态时，需要在事务中检查。为避免两个管理员并发操作导致管理员数量变成 0，建议使用 PostgreSQL 串行化事务，并在 Prisma 冲突时有限重试。

### 不信任前端

即使前端隐藏按钮，后端仍然必须检查：

- 操作者是不是管理员
- 目标用户是否存在
- 是否为自操作
- 是否会影响最后一个管理员
- 目标角色和状态是否合法

## 七、删除用户的处理

第一版不建议提供硬删除。

当前用户关联：

- AI 使用日志：用户删除时级联删除
- 文章对话：用户删除时级联删除
- RAG 问答：用户删除时将用户字段设为空

直接删除会造成使用数据和对话记录丢失，统计也会变化。

因此第一阶段使用：

```text
DISABLED = 已停用
```

替代删除。

如果以后确实需要删除，建议增加：

```prisma
deletedAt DateTime?
```

采用软删除或匿名化：

- 用户名改为 `deleted_<id>`
- 邮箱清空
- 密码改为随机不可登录值
- 状态设为 `DISABLED`
- 保留统计数据
- 清除个人身份信息

## 八、管理员操作审计

用户角色、状态和密码都属于敏感操作，建议同时增加审计表：

```prisma
model AdminAuditLog {
  id             Int      @id @default(autoincrement())
  operatorId     Int
  targetUserId   Int?
  action         String
  before         Json?
  after          Json?
  ip             String?
  userAgent      String?
  createdAt      DateTime @default(now())

  @@index([operatorId, createdAt])
  @@index([targetUserId, createdAt])
  @@index([action, createdAt])
}
```

操作类型可以先定义为：

```text
USER_CREATED
USER_ROLE_CHANGED
USER_STATUS_CHANGED
USER_PASSWORD_RESET
```

注意：

- 密码重置记录事件即可
- 绝对不能写入明文密码
- 也不要写入密码哈希
- IP 和 User-Agent 可复用现有工具取得

## 九、前端交互细节

为了降低误操作风险：

- 禁用用户前弹出确认框
- 管理员降级前弹出确认框
- 自己的禁用和降级按钮直接禁用，并显示原因
- 最后一个管理员的相关按钮禁用
- 重置密码使用独立弹窗，不放在普通编辑表单里
- 操作成功后只刷新当前用户或列表，不整页刷新
- API 返回 `409` 时展示具体原因
- 切换搜索或筛选条件后回到第 1 页
- 搜索输入增加约 300ms 防抖
- 分页、筛选条件同步到 URL

例如：

```text
/admin/users?page=2&search=zhang&role=USER&status=ACTIVE
```

这样刷新和返回页面时不会丢失条件。
